# 提醒桥接与未来任务落地

## 背景

NAS 运行日志已经确认：

- AngelHeart 会正常放行“明天早上 8 点提醒我开会”这类消息
- AstrBot 主脑这轮没有实际调用 `create_future_task`
- 最终只发出了“任务已锁”之类口头确认，未来任务列表没有新增

这说明故障不在秘书判断，而在“主模型是否真的调用未来任务工具”这一步。

## 本次实现

在不改现有秘书分析主链的前提下，新增一层“提醒/推送任务桥接器”：

1. 前台先识别是否为明确提醒请求或明确的定时推送/通知请求
2. 命中后按任务类型分流：
   - `提醒类`：走直发 `basic cron`
   - `推送/通知类`：继续走 `active_agent`
3. 成功则立即给用户明确确认
4. 失败则立即给用户明确失败原因
5. 这类消息不再继续交给主脑，避免再次出现“口头答应但没建任务”
6. 对明确的“列任务/删任务”语句直接调用 AstrBot future task 管理接口

## 设计原则

- 只拦截“明确包含提醒意图且能解析出时间”的消息
- 默认保守，不对普通闲聊做激进猜测
- 创建失败时优先显式报错，而不是继续让主模型自由发挥
- 使用 AstrBot 原生 future task / cron 机制，而不是插件内存里的临时定时器
- 对“必须送达”的提醒类任务优先保证送达，不把最后一步留给模型二次决定

## 配置

新增配置项：

- `reminder_future_task_enabled`

关闭后，这条桥接能力完全停用，消息恢复原有秘书/主脑处理流程。

## 并存策略

当前桥接器已经识别这些“不要覆盖”表达：

- `不需要覆盖`
- `不要覆盖`
- `不用覆盖`
- `别覆盖`
- `保留原任务`
- `保留旧任务`
- `两个任务并存`
- `任务并存`

这类表达当前会体现在确认文案里，明确告诉用户“不会覆盖旧任务”。桥接器本身只负责新增 future task，不会主动删旧任务。

## 任务管理

当前已支持：

- 列出当前会话下的 future task
- 按关键词删除当前会话下的 future task

当前还不支持：

- 没有明确旧任务标识时，自动“覆盖旧任务”
- 基于引用消息自动锁定具体 job_id

## 执行轨道

### 提醒类

例如：

- `1分钟后提醒我开户`
- `晚上11点提醒我睡觉`

这类任务会创建 `basic cron`，到点后插件直接发送提醒消息。

### 推送/通知类

例如：

- `每天早上7点推送江阴天气`
- `工作日8点通知我日报结果`

这类任务仍然创建 `active_agent`，保留动态内容获取和 agent 执行能力。

## 日志

新增固定前缀日志：

- `提醒桥接命中`
- `提醒桥接创建成功`
- `提醒桥接创建失败`
- `提醒桥接解析失败`

## 已知限制

- 当前还不支持“季度末”“月初第一个工作日”“每隔两天”“真正的双周循环”
- 当前“每隔两天”会映射为 cron 的 `*/2` 日规则，跨月边界不是严格意义上的固定 48 小时间隔
- 当前“季度末”“月初第一个工作日”会创建“下一次单次任务”，不会自动形成复杂长期循环
- 当前还不支持“真正的双周循环”“每隔两个月”“倒数第二个工作日”
- 当前中文数字时间支持有限，分钟主要覆盖 `半`、`十五`、`三十`、`四十五` 等常见写法
- 当前提醒正文提取偏保守，重点优先保证“建任务成功且不误伤”
