# 提醒任务直发双轨

## 背景

AstrBot 原生 `active_agent` future task 在到点后会唤醒主模型，再由模型决定是否调用 `send_message_to_user`。

这对开放式 agent 任务是合理的，但对“1分钟后提醒我开户”这类确定性提醒，可靠性不够：

- 任务已创建
- 到点已唤醒
- 但模型仍可能没有真正发消息

## 双轨方案

按任务类型分流：

- `提醒类任务`：走直发轨道
- `推送/通知/播报类任务`：继续走 agent 轨道

## 为什么这样分

### 适合直发

- `1分钟后提醒我开户`
- `晚上11点提醒我睡觉`
- `每周一提醒我开组会`

这类任务的核心要求是“必须送达”，不需要到点再思考。

### 适合 agent

- `每天早上7点推送江阴天气`
- `到点通知我最新日报结果`
- `下周一帮我总结群里重点再发给我`

这类任务本身带有动态信息获取或开放式执行，不适合降级成固定文本。

## 当前实现原则

- 默认开启提醒直发
- 只对 `request_kind == reminder` 生效
- `delivery` 任务仍使用 `active_agent`
- 直发任务使用 `basic cron job`
- 为了支持重启后恢复，插件启动后会回收并重新挂载自己的 reminder handler

## 风险边界

- 这不是替代 AstrBot 的 agent 设计，而是把“确定性交付”从概率型 agent 决策里剥离出来
- 对需要动态内容的任务，仍然保留 agent 轨道
